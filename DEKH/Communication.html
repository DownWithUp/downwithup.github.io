Communication

Last Update: 3/15/19

From a user mode program’s standpoint there are only two designated APIs to interact with kernel drivers, *ControlFile, and FilterSendMessage. Other methods for communication like shared memory regions or files could/may exist but are not discussed here.
Contrary to popular belief, it is possible to send two types of IRP_MJ_XXX requests.
IRP_MJ_DEVICE_CONTROL = 0xE from: DeviceIoControlFile -> NtDeviceIoControlFile -> Kernel IRP_MJ_FILE_SYSTEM_CONTROL = 0xD from: NtFsContolFile -> Kernel
A full list of major IRP codes can be found here: Click Me
It is up to the driver to decide on how to handle these request types. 0xE is specifically designed to be used for user mode interactions, while the lesser known 0xD isn’t commonly given a unique function. From personal experience 0xD is often implemented as an additional (or the only) unload subroutine.
The second common method of communication is usable if the driver calls FltCreateCommunicationPort and specifies a MessageNotifyCallback argument. A user mode program can use FilterConnectCommunicationPort and then FilterSendMessage.
The buffers sent from FilterSendMessage pre-filtered with ProbeForRead and ProbeForWrite (read for input and write for output) but may be unaligned and are vulnerable to race condition attacks because they reside directly in user mode address space. Source Link Of course, not all drivers call FltCreateCommunicationPort, or specify routines for handling IRP major functions. Some drivers live completly in kernel mode and don't interact with user mode. Checking a driver's API imports can be a quick method for determining what the function of the driver is and if it can communicate with user mode.
